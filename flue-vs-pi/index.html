Architecture explainer

Flue vs Pi

Short version: Pi is the broader agent/coding-agent stack, while Flue is a newer headless framework that reuses Pi’s lower-level engine pieces to make deployable, programmable agents.

What Flue is

Flue is a framework for building autonomous agents as application code. You write agent handlers in TypeScript, call init() to get a harness, then work through sessions, tools, sandboxes, skills, and HTTP deployment targets.

Verified in: flue/packages/runtime/src/harness.ts, session.ts, runtime/flue-app.ts
Pi mental model "Agent engine plus coding-agent product"

What Pi is

Pi is a monorepo with multiple layers: a provider abstraction, a generic agent loop, a session/runtime layer for coding work, and UI surfaces like the terminal app. The coding agent is the visible product, but it sits on reusable lower-level packages.

pi-coding-agent
CLI, sessions, slash commands, extensions, built-in coding tools
↓
pi-agent-core
Agent loop, tool execution, streaming lifecycle, steering/follow-up queues
↓
pi-ai
Unified model/provider layer across Anthropic, OpenAI, Google, Bedrock, etc.
Verified in: pi/packages/coding-agent/src/core/sdk.ts, pi/packages/agent/src/agent.ts, agent-loop.ts
The most important relationship: Flue is not just “similar to Pi.” It literally builds on Pi internals. In flue/packages/runtime/package.json, Flue depends on @mariozechner/pi-agent-core and @mariozechner/pi-ai. So Flue is effectively packaging Pi’s lower-level engine into a new framework-oriented developer experience.

Flue architecture

1. Agent-first API

The public abstraction is the agent handler. A handler gets a FlueContext, calls init(), and then works through a harness/session object rather than managing raw provider calls itself.

2. Harness/session split

Harness owns config + storage + environment. Session owns the conversation, prompt/skill/task/shell calls, history persistence, and compaction.

3. Deployment-aware runtime

Flue includes runtime machinery for Node and Cloudflare. Its Hono-based app layer exposes agent routes, run routes, and per-target execution behavior.

1
HTTP or CLI trigger hits an agent.
Agent routes are mounted by Flue’s runtime app layer.
2
init() builds a harness.
This wires together model defaults, tools, roles, session store, and sandbox.
3
harness.session() opens a durable conversation.
Sessions are keyed by instance/harness/session name and persisted via a store.
4
Prompt/skill/task calls drive the agent loop.
Flue wraps pi-agent-core’s Agent and injects built-in tools like read/write/edit/bash/task.
5
Sandbox abstracts execution.
Virtual bash is the default; local or remote/container connectors are pluggable.

Pi architecture

LayerWhat it doesWhy it matters
pi-ai Model registry, provider adapters, streaming protocol normalization, auth/key resolution. Makes the rest of Pi mostly provider-agnostic.
pi-agent-core Runs the agent loop: send messages, stream assistant output, execute tools, feed tool results back, continue until stop. This is the reusable brainstem both Pi and Flue rely on.
pi-coding-agent Adds coding workflows: file tools, session persistence, slash commands, extensions, prompt templates, system prompt building. This is the actual coding assistant product users interact with.
pi-tui Terminal UI library with differential rendering. Lets Pi ship a rich interactive CLI instead of only a headless API.
Pi’s center of gravity is operator experience. It is deeply optimized for an interactive coding session: sessions on disk, model switching, queued steering/follow-up messages, slash commands, extension hooks, session export, TUI, and RPC/SDK surfaces.

How they relate

Shared DNA

Flue reuses Pi’s engine instead of reinventing it. The strongest evidence is direct package dependency plus matching concepts:

  • assistant/tool-call loop
  • session state and message history
  • context compaction
  • provider abstraction
  • tool-based agent execution

Divergence in product shape

Pi grows “upward” into a coding-agent product and UI. Flue grows “outward” into a framework/runtime for deployable agents, sandboxes, connectors, and web-triggered workloads.

Best practical mental model

If you want to understand Flue internals → it helps to understand pi-agent-core and pi-ai, because that is where much of the underlying loop/provider machinery comes from.

One-sentence summaries

Flue: a headless agent application framework that turns Pi-style agent mechanics into deployable app infrastructure.

Pi: a modular agent stack whose most visible product is a powerful coding-agent CLI, backed by reusable core runtime and model layers.
